The Datadog SNMP check collects metrics from network devices by querying OIDs defined in SNMP profiles. A profile is a YAML file that maps a specific device make and model (identified by its `sysobjectid`) to a set of OID queries and metric definitions.

This tutorial covers the SNMP protocol concepts you need to understand, the profile file format, and the end-to-end workflow for building a new profile.

## SNMP concepts

### What is SNMP?

SNMP (Simple Network Management Protocol) is a protocol for monitoring [network devices](https://en.wikipedia.org/wiki/Networking_hardware). It uses UDP and supports a request/response model where the **SNMP manager** (the Datadog Agent) issues commands (`GET`, `GETNEXT`, `BULK`) to an **SNMP agent** (a network device).

SNMP exists in three versions:

| Version | Authentication | Transport layer |
|---|---|---|
| v1/v2c | Password (the _community string_) | Plain text only |
| v3 | Username/password | Support for packet signing and encryption |

### OIDs

An **OID** (Object Identifier) identifies a queryable quantity on an SNMP device — uptime, temperature, network traffic, and so on. OIDs are represented as dot-separated sequences of numbers, e.g. `1.3.6.1.2.1.1.1`.

OIDs are structured in a tree. There are two main sub-trees:

- **Generic OIDs** under `1.3.6.1.2.1.*` — applicable to all network devices.
- **Vendor-specific OIDs** under `1.3.6.1.4.1.*` — each vendor has its own sub-tree (`1.3.6.1.4.1.9.*` for Cisco, `1.3.6.1.4.1.2.*` for IBM, etc.).

Notable OIDs used during profile development:

| OID | Label | Description |
|---|---|---|
| `1.3.6.1.2.1.1.2` | `sysObjectId` | An OID whose value identifies the device make and model. Used for profile matching. |
| `1.3.6.1.2.1.1.1` | `sysDescr` | Human-readable description of the device. |
| `1.3.6.1.2.1.1.3` | `sysUpTimeInstance` | Device uptime. |

### MIBs

OIDs are grouped in modules called **MIBs** (Management Information Base). An MIB describes the hierarchy of a set of OIDs and assigns a human-readable label to each. OID resolution is the process of finding an OID from its label.

Useful MIB resources:

- [MIB Discovery](http://cric.grenoble.cnrs.fr/Administrateurs/Outils/MIBS/) — a search engine for OIDs and MIBs.
- [Circitor MIB files repository](http://circitor.fr/Mibs/Mibs.php) — download `.mib` files.
- [SNMP Labs MIB repository](https://github.com/DataDog/mibs.snmplabs.com/tree/master/asn1) — the MIBs validated against by the `pysnmp-mibs` library used by the SNMP Python check.

## What is an SNMP profile?

An SNMP profile is a YAML file that provides out-of-the-box monitoring for a specific network device make and model. It defines:

- The `sysobjectid` that identifies which devices the profile applies to.
- The metrics to collect, as OID references.
- Optional metric tags and metadata.

The Datadog Agent matches a discovered device's `sysObjectId` against all available profiles and automatically applies the matching one.

## Profile file format

A profile has the following top-level structure:

```yaml
sysobjectid: <x.y.z...>

# extends:
#   <Optional list of base profiles to extend from...>

metrics:
  # <List of metrics to collect...>

# metric_tags:
#   <List of tags to apply to all collected metrics>
```

### `sysobjectid`

The `sysobjectid` field identifies which devices use this profile. It can be a single OID, a wildcard, or a list:

```yaml
# Specific device model
sysobjectid: 1.3.6.1.4.1.232.9.4.10

# Wildcard covering multiple models
sysobjectid: 1.3.6.1.131.12.4.*

# List of OIDs and wildcards
sysobjectid:
  - 1.3.6.1.131.12.4.*
  - 1.3.6.1.4.1.232.9.4.10
```

### `extends`

Use `extends` to inherit metrics and tags from base profiles. All device profiles should extend from `_base.yaml`:

```yaml
extends:
  - _base.yaml
  - _generic-if.yaml  # Include basic metrics from IF-MIB
```

### `metrics` — symbol metrics

A symbol metric collects a single scalar OID value. Specify the `MIB`, and the `OID` and `name` of the symbol:

```yaml
metrics:
  - MIB: ISILON-MIB
    symbol:
      OID: 1.3.6.1.4.1.12124.1.1.2
      name: clusterHealth
  - MIB: ISILON-MIB
    symbol:
      OID: 1.3.6.1.4.1.12124.1.2.1.1
      name: clusterIfsInBytes
```

<Warning>
  Symbol metrics from the same MIB must be listed as separate `metrics` entries. A single entry cannot have a list under `symbol`.
</Warning>

### `metrics` — table metrics

Table metrics collect values across all rows of an SNMP table. Specify the table OID and each column symbol you want to collect:

```yaml
metrics:
  - MIB: CISCO-PROCESS-MIB
    table:
      OID: 1.3.6.1.4.1.9.9.109.1.1.1
      name: cpmCPUTotalTable
    symbols:
      - OID: 1.3.6.1.4.1.9.9.109.1.1.1.1.12
        name: cpmCPUMemoryUsed
    metric_tags:
      # ...
```

Table metrics require `metric_tags` to distinguish rows from one another.

### Table metric tagging

<AccordionGroup>
  <Accordion title="Tag using a column from the same table">
    The most common approach — use a descriptive column from the same table as a tag value:

    ```yaml
    metrics:
      - MIB: IF-MIB
        table:
          OID: 1.3.6.1.2.1.2.2
          name: ifTable
        symbols:
          - OID: 1.3.6.1.2.1.2.2.1.14
            name: ifInErrors
        metric_tags:
          - tag: interface
            symbol:
              OID: 1.3.6.1.2.1.2.2.1.2
              name: ifDescr
    ```
  </Accordion>

  <Accordion title="Tag using a column from a different table">
    When the tag source is in a different table with the same row indexes:

    ```yaml
    metrics:
      - MIB: CISCO-IF-EXTENSION-MIB
        metric_type: monotonic_count
        table:
          OID: 1.3.6.1.4.1.9.9.276.1.1.2
          name: cieIfInterfaceTable
        symbols:
          - OID: 1.3.6.1.4.1.9.9.276.1.1.2.1.1
            name: cieIfResetCount
        metric_tags:
          - MIB: IF-MIB
            symbol:
              OID: 1.3.6.1.2.1.31.1.1.1.1
              name: ifName
            table: ifXTable
            tag: interface
    ```
  </Accordion>

  <Accordion title="Tag using an index value">
    Use an index position when the row index itself is meaningful. `index: 1` refers to the first digit of the row index:

    ```yaml
    metrics:
      - MIB: CISCO-FIREWALL-MIB
        table:
          OID: 1.3.6.1.4.1.9.9.147.1.2.2.2
          name: cfwConnectionStatTable
        symbols:
          - OID: 1.3.6.1.4.1.9.9.147.1.2.2.2.1.5
            name: cfwConnectionStatValue
        metric_tags:
          - index: 1
            tag: service_type
          - index: 2
            tag: stat_type
    ```
  </Accordion>

  <Accordion title="Map values to tag strings">
    Map numeric OID values to human-readable tag strings (available in Agent 7.45+):

    ```yaml
    metrics:
      - MIB: IP-MIB
        table:
          OID: 1.3.6.1.2.1.2.2
          name: ifTable
        symbols:
          - OID: 1.3.6.1.2.1.2.2.1.10
            name: ifInOctets
        metric_tags:
          - tag: if_type
            symbol:
              OID: 1.3.6.1.2.1.2.2.1.3
              name: ifType
            mapping:
              1: other
              2: regular1822
              3: hdh1822
              4: ddn-x25
              29: ultra
    ```
  </Accordion>
</AccordionGroup>

### Metric types

By default, the Datadog metric type is inferred from the SNMP type:

| SNMP type | Inferred metric type |
|---|---|
| `Counter32` | `rate` |
| `Counter64` | `rate` |
| `Gauge32` | `gauge` |
| `Integer` | `gauge` |
| `Integer32` | `gauge` |
| `CounterBasedGauge64` | `gauge` |

To override the inferred type, set `metric_type` explicitly:

| Forced type | Description |
|---|---|
| `gauge` | Submit as a gauge. |
| `rate` | Submit as a rate. |
| `percent` | Multiply by 100 and submit as a rate. |
| `monotonic_count` | Submit as a monotonic count. |
| `monotonic_count_and_rate` | Submit two metrics: one monotonic count and one rate (suffixed with `.rate`). |
| `flag_stream` | Submit each bit position of a flag stream as a separate metric with value `0` or `1`. |

Example of forcing `monotonic_count` on a counter that represents a total:

```yaml
metrics:
  - MIB: TCP-MIB
    symbol:
      OID: 1.3.6.1.2.1.6.5
      name: tcpActiveOpens
      metric_type: monotonic_count
```

### Symbol modifiers

Use `extract_value` to parse a numeric value out of a string OID value (for example, strip a unit suffix):

```yaml
metrics:
  - MIB: MY-MIB
    symbol:
      OID: 1.2.3.4.5.6.7
      name: temperature
      extract_value: '(\d+)C'
```

Use `scale_factor` to convert units. For example, to convert kilobytes to bytes:

```yaml
metrics:
  - MIB: AIRESPACE-SWITCHING-MIB
    symbol:
      OID: 1.3.6.1.4.1.14179.1.1.5.3
      name: memory.free
      scale_factor: 1000
```

Use `format: mac_address` or `format: ip_address` in `metric_tags` to format binary-encoded addresses into human-readable strings:

```yaml
metric_tags:
  - symbol:
      OID: 1.3.6.1.4.1.29671.1.1.4.1.1
      name: devMac
      format: mac_address
    tag: mac_address
```

## Build a profile from scratch

<Steps>
  <Step title="Research the device">
    Before writing any YAML, gather the following information:

    - The device's `sysobjectid`. This is available from the device documentation or by querying `sysObjectId` (`1.3.6.1.2.1.1.2`) on a live device.
    - Which MIBs the device supports. Device vendors typically publish MIB packages in their support portals.
    - What metrics are worth collecting (10–40 is a good target). Look for health status gauges, network traffic counters (bytes in/out, errors), CPU and memory usage, temperature sensors, power supply status, and storage.

    Base profiles like `_generic-if.yaml` and `_generic-ip.yaml` already cover common OIDs from standard MIBs. Extend from them to avoid duplicating metric definitions.
  </Step>

  <Step title="Create the profile file">
    Add a `.yaml` file with the `sysobjectid` and at least one metric to start:

    ```yaml
    sysobjectid: 1.3.6.1.4.1.232.9.4.10

    extends:
      - _base.yaml

    metrics:
      - MIB: CPQHLTH-MIB
        symbol:
          OID: 1.3.6.1.4.1.232.6.2.8.1.0
          name: cpqHeSysUtilLifeTime
    ```
  </Step>

  <Step title="Write a unit test">
    Add a unit test in `test_profiles.py` to verify the metric is collected when the profile is active. The test will fail at first because there's no simulation data yet — that's expected:

    ```python
    def test_hp_ilo4(aggregator):
        run_profile_check('hp_ilo4')

        common_tags = common.CHECK_TAGS + ['snmp_profile:hp-ilo4']

        aggregator.assert_metric(
            'snmp.cpqHeSysUtilLifeTime',
            metric_type=aggregator.MONOTONIC_COUNT,
            tags=common_tags,
            count=1
        )
        aggregator.assert_all_metrics_covered()
    ```

    Run it to confirm the failure:

    ```bash
    ddev test -k test_hp_ilo4 snmp:py38
    ```
  </Step>

  <Step title="Add simulation data">
    Create a `.snmprec` file in `snmp/tests/compose/data/` named after the community string you passed to `run_profile_check()`:

    ```bash
    touch snmp/tests/compose/data/hp_ilo4.snmprec
    ```

    Each line in a `.snmprec` file follows the format `<OID>|<type>|<value>`. Add an entry for the `sysObjectId` (type `6` = OID) and one for each metric OID (type `2` = integer):

    ```
    1.3.6.1.2.1.1.2.0|6|1.3.6.1.4.1.232.9.4.10
    1.3.6.1.4.1.232.6.2.8.1.0|2|1051200
    ```

    <Warning>
      Lines in `.snmprec` files must be sorted in lexicographic order. Run `sort -V /path/to/profile.snmprec` to sort them.
    </Warning>

    For table metrics, add one copy of each column OID per row by appending the row index:

    ```
    1.3.6.1.4.1.6.13.1.6.0|2|1051200
    1.3.6.1.4.1.6.13.1.6.1|2|1446
    1.3.6.1.4.1.6.13.1.6.2|2|23
    ```

    To generate table simulation data automatically, use the `mib2dev.py` tool from `snmpsim`:

    ```bash
    pip install snmpsim
    mib2dev.py --mib-module=IF-MIB \
      --start-oid=1.3.6.1.2.1.2.2 \
      --stop-oid=1.3.6.1.2.1.2.3 \
      --table-size=4 > /path/to/mytable.snmprec
    ```
  </Step>

  <Step title="Run the test again">
    With simulation data in place, the test should pass:

    ```bash
    ddev test -k test_hp_ilo4 snmp:py38
    ```

    Expected output:

    ```
    tests/test_profiles.py::test_hp_ilo4 PASSED                 [100%]

    1 passed, 107 deselected in 9.87s
    ```
  </Step>

  <Step title="Expand the profile">
    With the basic workflow confirmed, add more metrics to the profile, expand the unit test, and add corresponding simulation data. Repeat until you've covered the metrics identified in your research phase.
  </Step>
</Steps>

## Test profiles against a simulated device

The SNMP integration includes a simulated SNMP device you can use for local testing.

### Start the test environment

```bash
ddev env start snmp [...]
```

### Configure an instance to use your profile

```yaml
init_config:
  profiles:
    cisco_nexus:
      definition_file: cisco-nexus.yaml

instances:
  - community_string: cisco_nexus
    ip_address: <IP_ADDRESS_OF_SNMP_CONTAINER>
    profile: cisco_nexus
    name: localhost
    port: 1161
```

The `community_string` must match the `.snmprec` (or `.snmpwalk`) filename. To find the container's IP address:

```bash
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' dd-snmp
```

### Query the device directly

With Net-SNMP tools installed, you can issue queries directly to the simulated device.

<Tabs>
  <Tab title="GET query">
    Retrieve a single OID value:

    ```console
    $ snmpget -v 2c -c public -IR 127.0.0.1:1161 system.sysDescr.0
    SNMPv2-MIB::sysDescr.0 = STRING: Linux 41ba948911b9 4.9.87-linuxkit-aufs ...
    ```

    If the command fails, try the numeric OID form:

    ```console
    $ snmpget -v 2c -c public -IR 127.0.0.1:1161 iso.3.6.1.2.1.1.1.0
    ```
  </Tab>
  <Tab title="Table query">
    Retrieve all rows of a table:

    ```console
    $ snmptable -v 2c -c public -IR -Os 127.0.0.1:1161 hrStorageTable
    ```
  </Tab>
  <Tab title="Walk query">
    Retrieve all OIDs in a sub-tree. The `-ObentU` options print output in a format compatible with `.snmpwalk` simulation files:

    ```console
    $ snmpwalk -v 2c -c public -ObentU 127.0.0.1:1161 1.3.6.1.2.1.1
    .1.3.6.1.2.1.1.1.0 = STRING: Linux 41ba948911b9 ...
    .1.3.6.1.2.1.1.2.0 = OID: .1.3.6.1.4.1.8072.3.2.10
    .1.3.6.1.2.1.1.3.0 = 4226041
    [...]
    ```

    You can save this output as a `.snmpwalk` file and use it as simulation data by setting `community_string` to the filename (without extension).
  </Tab>
</Tabs>

### Use tcpdump to inspect SNMP traffic

To see the exact SNMP requests and responses the Agent sends, run `tcpdump` in a separate shell:

```bash
tcpdump -vv -nni lo0 -T snmp host localhost and port 161
```

Then trigger a collection. You'll see output like:

```
127.0.0.1.59540 > 127.0.0.1.1161: { SNMPv2c C="cisco-nexus" { GetRequest(28) R=1921760388 .1.3.6.1.2.1.1.2.0 } }
127.0.0.1.1161 > 127.0.0.1.59540: { SNMPv2c C="cisco-nexus" { GetResponse(40) R=1921760388 .1.3.6.1.2.1.1.2.0=.1.3.6.1.4.1.9.12.3.1.3.1.2 } }
```

To install `snmpwalk` and `tcpdump` inside the Docker Agent container:

```bash
apt update && apt install -y snmp tcpdump
```

## Browse locally installed MIBs

Use `snmptranslate` from Net-SNMP to list all OIDs in a MIB along with their labels:

```console
$ snmptranslate -m IF-MIB -Tz -On > out.log
$ cat out.log
"sysDescr"    "1.3.6.1.2.1.1.1"
"sysObjectID" "1.3.6.1.2.1.1.2"
"sysUpTime"   "1.3.6.1.2.1.1.3"
[...]
```

<Tip>
  Use the `-M <DIR>` option to point `snmptranslate` at a directory containing MIBs you've downloaded but not yet installed. Use `-Tp` for tree-formatted output.
</Tip>

## Generate a profile from MIB files

You can use `ddev` to generate a profile scaffold from a collection of ASN1 MIB files:

```bash
ddev meta snmp generate-profile-from-mibs --help
```

The tool accepts a list of MIB files and copies a list of metric definitions to the clipboard. Use the `-f` option to pass a YAML filter file that restricts which MIB nodes are included:

```yaml
RFC1213-MIB:
  - system
  - interfaces
  - ip
CISCO-SYSLOG-MIB: []
SNMP-FRAMEWORK-MIB:
  - snmpEngine
```

Use the `-a` option to pass alias definitions that map index columns to more descriptive tag names.
