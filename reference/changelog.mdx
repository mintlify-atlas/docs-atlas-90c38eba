## Overview

Every change to Python code inside a `datadog_checks` subdirectory requires a changelog entry. Changelog entries are **not** required for changes to tests or assets alone.

<Note>
  Changelog files **must** be created using the `ddev release changelog new` command. Do not create or edit changelog files manually.
</Note>

## The three change types

Each changelog entry has a type that determines the semantic version bump applied at release time:

| Type | Version bump | When to use |
|------|-------------|-------------|
| `fixed` | Patch (1.0.0 → 1.0.1) | Bug fixes |
| `added` | Minor (1.0.0 → 1.1.0) | New features or non-trivial refactors |
| `changed` | Major (1.0.0 → 2.0.0) | Breaking changes or significant modifications |

## Where changelog files live

Changelog entries for `integrations-core` are stored as individual files inside the integration's `changelog.d/` directory. Each file is named after the pull request number and the change type:

```
<INTEGRATION>/
└── changelog.d/
    ├── 12345.added
    ├── 12346.fixed
    └── 12347.changed
```

These files accumulate in the `changelog.d/` directory as PRs are merged. At release time, they are automatically consolidated into the integration's `CHANGELOG.md` under a new version section.

## Creating a changelog entry

Use the `ddev release changelog new` command:

```bash
ddev release changelog new <TYPE> <INTEGRATION> -m "<MESSAGE>"
```

<CodeGroup>

```bash Bug fix
ddev release changelog new fixed sqlserver \
  -m "Fix a bug where tempdb is wrongly excluded from database files metrics."
```

```bash New feature
ddev release changelog new added kafka_consumer \
  -m "Bump OpenSSL in confluent-kafka to 3.4.1 on Windows."
```

```bash Breaking change
ddev release changelog new changed postgres \
  -m "Update configuration options for connection pooling."
```

</CodeGroup>

The command creates a file at `<INTEGRATION>/changelog.d/<PR_NUMBER>.<TYPE>` containing your message. Commit this file alongside your code changes and include it in your pull request.

## Skipping the changelog

If your change does not affect shipped Agent behavior — for example, a documentation fix or a CI configuration update — you can skip the changelog requirement by applying the `changelog/no-changelog` label to your pull request. This suppresses the CI check that enforces changelog entries.

## How changelogs are released

<Steps>
  <Step title="Entries accumulate in changelog.d/">
    As PRs are merged, `changelog.d/<PR_NUMBER>.<TYPE>` files pile up inside each integration's directory.
  </Step>
  <Step title="Release tooling consolidates entries">
    When an integration is released, the release tooling reads all files in `changelog.d/`, sorts and groups them by type, and writes them into the `CHANGELOG.md` under a new version and date header.
  </Step>
  <Step title="changelog.d/ is cleared">
    After consolidation, the individual files in `changelog.d/` are removed.
  </Step>
</Steps>

## Reading CHANGELOG.md

Each integration has its own `CHANGELOG.md` at the root of its directory. The format follows the [Keep a Changelog](https://keepachangelog.com) convention:

```markdown
# CHANGELOG - Foo Integration

## Unreleased

***Changed***:

* Made a breaking change ([#9000](https://github.com/DataDog/repo/pull/9000))

    Additional context about the breaking change and migration steps.

***Added***:

* Added a cool new feature ([#42](https://github.com/DataDog/repo/pull/42))

## 1.2.3 / 2081-04-01

***Fixed***:

* Fixed a bug in metric collection ([#38](https://github.com/DataDog/repo/pull/38))
```

### Formatting rules

<AccordionGroup>
  <Accordion title="Spacing">
    Place a blank line between each of the following sections:

    - The file header (`# CHANGELOG - Integration Name`)
    - The `## Unreleased` header
    - Each version/date header (`## 1.2.3 / 2024-01-01`)
    - Each change type header (`***Fixed***:`, `***Added***:`, etc.)
    - The list of changes under each type

    Do **not** add blank lines between individual bullet points within a single change type section.
  </Accordion>
  <Accordion title="Version header format">
    Version headers follow this pattern:

    ```
    ## <VERSION> / <YYYY-MM-DD>
    ```

    The date must use the `YYYY-MM-DD` format with no spaces. The Agent version is optional but may be appended. The first header after the file title may be `## Unreleased`.
  </Accordion>
  <Accordion title="Change type headers">
    Change type headers must be bold and italic, in uppercase, and terminated with a colon:

    ```
    ***Changed***:
    ***Added***:
    ***Fixed***:
    ```

    The headers must appear in the following order within a version section: `Changed`, `Added`, `Deprecated`, `Removed`, `Fixed`, `Security`.
  </Accordion>
  <Accordion title="Entry format">
    Each changelog entry is a bullet point using an asterisk. The first line must end with a link to the pull request in which the change occurred:

    ```markdown
    * Fixed incorrect metric name for disk I/O ([#1234](https://github.com/DataDog/integrations-core/pull/1234))
    ```

    You may add indented additional context below the first line:

    ```markdown
    * Fixed incorrect metric name for disk I/O ([#1234](https://github.com/DataDog/integrations-core/pull/1234))

        The metric was previously submitted as `disk.io_read` but the correct
        name is `disk.read_bytes`. Update any monitors or dashboards that use
        the old name.
    ```

    To automatically apply the PR link suffix to manually written entries, run:

    ```bash
    ddev release changelog fix
    ```
  </Accordion>
</AccordionGroup>

## Finding a changelog

Each integration's changelog is at `<INTEGRATION_NAME>/CHANGELOG.md` in the repository. For example:

```
postgres/CHANGELOG.md
snmp/CHANGELOG.md
vsphere/CHANGELOG.md
```

You can also view the agent-level changelog — which lists all integration updates shipped with each Agent release — at `AGENT_CHANGELOG.md` in the root of the repository. This file groups integration updates by Agent version and links to each integration's own `CHANGELOG.md` for details.

```markdown
## Datadog Agent version 7.77.0

### New Integrations
* Cisco ASA [1.0.0](...)

### Integration Updates
* PostgreSQL [18.3.0](...)
* Elasticsearch [9.3.0](...)
```
