---
title: "End-to-end testing"
description: "Run E2E tests against a live Datadog Agent using ddev env — start environments, choose Agent versions, test local code changes, invoke the Agent directly, and debug interactively."
---

Any integration that uses the ddev [pytest plugin](/ddev/plugins) in its test suite supports end-to-end testing on a live Datadog Agent. The entrypoint for E2E management is the `ddev env` command group.

In E2E mode, ddev spins up the integration's Docker services and a Datadog Agent container side by side. The Agent runs real check executions against the live service, so you can verify behavior that unit tests can't catch.

## Typical E2E workflow

<Steps>

<Step title="Choose an Agent version (optional)">

By default, ddev uses a custom nightly build of the Agent (`datadog/agent-dev:master-py3`) that is rebuilt on every commit to the Datadog Agent repository.

To use a specific Agent version, pass the `--agent` / `-a` option with any valid Docker image:

```bash
ddev env start postgres py3.9-14.0 --agent datadog/agent:7.47.0
```

</Step>

<Step title="Test local code changes (optional)">

By default, the integration version shipped with the chosen Agent is used. To test your local changes in real time, add the `--dev` flag:

```bash
ddev env start postgres py3.9-14.0 --dev
```

With `--dev`, ddev mounts and installs your local integration directory into the Agent container. Any code changes or branch switches in that directory are propagated automatically — no restart required.

If you also modified the [base package](/base/overview), add `--base` to mount it as well (`--base` implies `--dev`):

```bash
ddev env start postgres py3.9-14.0 --base
```

</Step>

<Step title="Run E2E tests">

Run tests marked as E2E against the live Agent using `ddev env test`:

```bash
ddev env test postgres py3.9-14.0
```

This is similar to `ddev test` but only runs tests marked as E2E. Use `--dev` here too if you started the environment with `--dev`:

```bash
ddev env test --dev postgres py3.9-14.0
```

</Step>

<Step title="Debug interactively">

Start an interactive [pdb](https://docs.python.org/3/library/pdb.html) debugging session using the `--breakpoint` / `-b` option. Pass an integer representing the line number at which to break:

```bash
ddev env agent postgres py3.9-14.0 check -b 0
```

<Warning>
  The line number must be within the integration's `check` method.
</Warning>

</Step>

<Step title="Reload after config changes">

Test runs and manual check invocations always reflect the current state of your code and config file. However, if you want to see the effect of changes in the Datadog app (for example, new metrics appearing on a dashboard), you need to reload the environment:

```bash
ddev env reload postgres py3.9-14.0
```

</Step>

<Step title="Stop the environment">

When you're done, stop the environment to shut down all containers:

```bash
ddev env stop postgres py3.9-14.0
```

<Note>
  Any environment that hasn't been explicitly stopped continues to appear as active in `ddev env show` output, even after system restarts.
</Note>

</Step>

</Steps>

## Recreate from scratch

Add `--recreate` to tear down and rebuild both the hatch environment and Docker containers/volumes:

```bash
ddev env test --dev --recreate postgres py3.9-14.0
```

Under the hood, `--recreate` runs `docker compose down --volumes` followed by `docker compose up -d --force-recreate`. Use this when a stale volume or environment state is causing unexpected test failures.
